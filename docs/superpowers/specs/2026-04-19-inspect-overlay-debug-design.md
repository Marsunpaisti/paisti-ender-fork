# Inspect Overlay Debug Expansion Design

**Goal:** Extend the existing Ender inspect tooltip so `Shift` inspect exposes richer gob debugging data without making normal inspect noisy.

## Scope

This change only affects the existing inspect tooltip path used by inspect mode.

In scope:

- preserving the current non-`Shift` inspect line
- expanding `Shift` inspect for gobs with additional debug information
- using the existing tooltip plumbing already routed through `CustomCursors.inspect(...)` and `Gob.inspect(boolean full)`

Out of scope:

- new keybinds or cursor modes
- a separate inspector window
- deep recursive dumps of object internals
- tile-inspect changes beyond current behavior

## Current Behavior

Inspect mode is triggered via `Action.TOGGLE_INSPECT` and handled through `CustomCursors`.

- `src/haven/Action.java`
- `src/me/ender/CustomCursors.java`
- `src/haven/MapView.java`

For hovered gobs, inspect mode currently shows `gob.inspect(map.fullTip)`. `MapView.fullTip` tracks whether `Shift` is held. `Gob.inspect(boolean full)` currently renders:

- `resid [sdt]` always
- material/customization information only when `full == true`

Relevant file:

- `src/haven/Gob.java`

## Chosen Approach

Implement the feature by extending `Gob.inspect(boolean full)`.

Why this approach:

- it is the smallest correct change
- it preserves the existing inspect mode contract
- it keeps the extra debug payload behind the existing `Shift` path
- it avoids introducing new UI or new state management

Rejected alternatives:

- separate inspector window: better UI, much larger scope
- new helper subsystem: cleaner abstraction, unnecessary for this targeted feature

## Desired Behavior

### Normal Inspect

Normal inspect remains unchanged:

- first line: `<resid> [<sdt-decimal>]`

This keeps the tooltip lightweight during casual use.

### Shift Inspect

When `full == true`, append a compact developer-oriented dump for gob tooltips.

The debug payload should include:

- gob id
- `sdt` decimal value
- `sdt` hexadecimal value
- raw `sdt` bytes
- numeric `sdt` bit breakdown
- gob attribute class names
- gob overlay summary
- existing material/customization info if present

The tooltip should remain text-only and line-oriented.

## Data Definitions

### Gob Id

Use the gob runtime id (`gob.id`).

### `sdt` Decimal

Use the existing `gob.sdt()` numeric interpretation.

### `sdt` Hex

Format the same numeric `sdt` value in hexadecimal.

### Raw `sdt` Bytes

Use `gob.sdtm()` to clone the state-data `Message`, then read the remaining bytes and render them in hex byte form.

If `sdtm()` is `null`, show an empty or explicit no-data representation.

### Bit Breakdown

Render the numeric `sdt` as a binary string for quick flag inspection.

This is intentionally generic. The tooltip should not try to globally interpret what each bit means, since `sdt` semantics are resource-specific.

### Attributes

Summarize the gob's current attribute set by class name only.

This should be a flat list of attached `GAttrib` implementation class names. Do not recursively dump field contents.

### Overlays

Summarize gob overlays in a lightweight way.

Prefer a list of overlay resource ids where available; otherwise fall back to overlay ids or class names. The goal is quick inspection of "what overlays are attached" rather than full internal state.

### Materials

Retain the existing material/customization dump generated by `CustomizeVarMat.formatMaterials(this)`.

## Formatting

Use concise labels and stable formatting. Proposed output shape:

```text
gfx/terobjs/trees/laurel [2]
id: 123456789
sdt-dec: 2
sdt-hex: 0x2
sdt-bytes: 02
sdt-bits: 10
attribs: ResDrawable, GobHealth, GeneralGobInfo
overlays: gfx/fx/eq, gfx/terobjs/trees/laurelleaf
materials:
...
```

Formatting goals:

- one fact per line
- omit empty sections where possible
- do not reflow into paragraph text

## Error Handling

- If a value is unavailable, skip that line or show a compact sentinel such as `none`.
- If resource resolution for overlays is still loading, do not fail the entire tooltip.
- The inspect tooltip must remain safe under partial gob loading.

## Performance Constraints

This code runs on hover, so the implementation should avoid expensive recursive introspection.

Acceptable work:

- cloning `sdt`
- formatting a short byte array
- iterating current gob attributes
- iterating current overlay list

Avoid:

- loading unrelated resources just to enrich the tooltip
- reflective field dumping
- building very large strings for every frame when data is absent

## Files To Modify

- `src/haven/Gob.java`

Possible supporting touch if needed, but not preferred:

- none

## Testing Strategy

Manual verification is sufficient for this feature.

Verify in-game:

1. normal inspect still shows the current short tooltip
2. holding `Shift` expands the tooltip for gobs
3. expanded tooltip includes id, `sdt` decimal, hex, bytes, bits, attributes, and overlays
4. expanded tooltip does not crash on partially loaded gobs
5. tile inspect behavior is unchanged

## Risks

- overlay/resource summaries may surface internal names that are noisy
- some gobs may have many attributes or overlays, producing a tall tooltip
- `sdt` binary output is generic and may tempt overinterpretation even though meanings are resource-specific

## Mitigations

- keep the heavy payload behind `Shift`
- keep summaries shallow
- preserve the current first line as the anchor for quick reading

## Open Decisions Resolved

- Full debug payload should be available: yes
- Full debug payload should be behind `Shift`: yes
- Separate inspector UI: no
- Deep reflective dump: no
